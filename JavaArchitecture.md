# Java Execution Flow, JVM, JDK, and JRE Notes

# Java Execution Flow

``` text
.java File (human-readable source code)
            ↓
     javac (Compiler)
            ↓
.class File (Bytecode)
            ↓
           JVM
            ↓
    Interpreter + JIT Compiler
            ↓
      Machine Code (0s and 1s)
            ↓
         Hardware
```

## What is a `.java` File?

-   A `.java` file contains source code written by developers.
-   It is written in a human-readable format using Java syntax.
-   Machines cannot directly understand this code.

Example:

``` java
public class Main {
    public static void main(String[] args) {
        System.out.println("Hello World");
    }
}
```

------------------------------------------------------------------------

## What is Machine Code?

-   Machine code consists of binary instructions made up of **0s and
    1s**.
-   It is the only language understood directly by the processor (CPU).
-   Since humans cannot conveniently write programs using binary
    instructions, programming languages such as Java are used.

------------------------------------------------------------------------

## Java Compilation Process

### Step 1: Source Code to Bytecode

The Java compiler (`javac`) converts the `.java` file into a `.class`
file.

``` text
Main.java
    ↓
 javac Main.java
    ↓
Main.class
```

Important points:

-   The compiler **does not generate machine code**.
-   The compiler generates **bytecode**.
-   Bytecode is an intermediate representation of the program.
-   Bytecode is designed to be executed by the JVM.

------------------------------------------------------------------------

## What is Bytecode?

-   Bytecode is platform-neutral code generated by the Java compiler.
-   It has the `.class` extension.
-   It is not understandable by the CPU directly.
-   It requires the JVM for execution.

------------------------------------------------------------------------

## JVM Execution Process

After the `.class` file is generated, the following activities happen
during runtime:

``` text
Class Loader
      ↓
Bytecode Verifier
      ↓
Interpreter
      ↓
JIT Compiler (for frequently executed code)
      ↓
Machine Code
      ↓
Hardware
```

------------------------------------------------------------------------

# Class Loader

The Class Loader subsystem is responsible for loading classes into
memory.

It performs three major activities:

## 1. Loading

During loading:

-   The required `.class` files are located and read.
-   Binary data from the class files is brought into memory.
-   An internal representation of the class is created.

------------------------------------------------------------------------

## 2. Linking

Linking consists of three phases.

### Verification

-   JVM checks whether the bytecode follows Java specifications.
-   It ensures the code has not been corrupted.
-   It prevents illegal operations.

### Preparation

-   Memory is allocated for static variables.
-   Default values are assigned.

Example:

``` java
static int count;
```

Default value assigned during preparation:

``` java
count = 0;
```

### Resolution

-   Symbolic references are replaced with direct references.
-   JVM identifies actual memory locations of referenced classes,
    methods, and fields.

------------------------------------------------------------------------

## 3. Initialization

During initialization:

-   Static variables receive their explicitly assigned values.
-   Static blocks are executed.

Example:

``` java
static int count = 10;

static {
    System.out.println("Static block executed");
}
```

------------------------------------------------------------------------

# Bytecode Verifier

The Bytecode Verifier checks:

-   The bytecode format is valid.
-   Instructions do not violate JVM rules.
-   Access permissions are correct.
-   Stack operations are legal.
-   No unsafe operations exist.

This improves security and reliability.

------------------------------------------------------------------------

# Interpreter

The interpreter:

-   Reads bytecode instruction by instruction.
-   Converts each instruction into machine code.
-   Executes the instructions immediately.

Example:

If a method is executed only once, the interpreter handles it
efficiently.

Disadvantage:

-   Frequently executed methods get interpreted repeatedly.
-   Repeated interpretation can reduce performance.

------------------------------------------------------------------------

# JIT (Just-In-Time) Compiler

To improve performance, JVM uses JIT compilation.

JIT identifies frequently executed code sections, called **hot spots**.

It compiles those sections directly into machine code.

Advantages:

-   Eliminates repeated interpretation.
-   Improves execution speed.
-   Optimizes heavily used methods.

Therefore:

``` text
Interpreter → Good for infrequently executed code.

JIT Compiler → Good for repeatedly executed code.
```

------------------------------------------------------------------------

# Garbage Collector

Garbage Collector is a part of JVM.

Responsibilities:

-   Identifies unused objects.
-   Frees heap memory occupied by unused objects.
-   Reduces memory leaks.
-   Automatically manages memory.

Developers do not explicitly free memory like in C or C++.

------------------------------------------------------------------------

# Platform Independence

Platform independence means:

> Java programs can run on different operating systems without
> recompilation.

Reason:

-   The compiler generates bytecode.
-   Bytecode can run on any system having a compatible JVM.

Example:

``` text
Windows → javac → Bytecode
                    ↓
             Linux JVM → Executes
                    ↓
             macOS JVM → Executes
```

------------------------------------------------------------------------

## Why C/C++ Are Platform Dependent

In C/C++:

``` text
Source Code
     ↓
Compiler
     ↓
Machine Code (.exe)
```

-   Machine code generated for Windows cannot directly run on Linux.
-   Separate compilation is required for each platform.

------------------------------------------------------------------------

# Important Interview Statement

Java is platform independent because:

-   Java compiler generates platform-independent bytecode.

JVM is platform dependent because:

-   Different operating systems require different JVM implementations.

Therefore:

``` text
Java → Platform Independent

JVM → Platform Dependent
```

------------------------------------------------------------------------

# JDK (Java Development Kit)

JDK provides an environment to develop and run Java applications.

It contains:

-   JRE
-   Java compiler (`javac`)
-   Java Archiver (`jar`)
-   Javadoc tool
-   Debugging tools
-   Monitoring tools
-   Development utilities

------------------------------------------------------------------------

## Important JDK Tools

### javac

-   Java compiler.
-   Converts `.java` files into `.class` files.

### jar

-   Java Archiver tool.
-   Packages multiple class files and resources into a JAR file.

### javadoc

-   Generates API documentation from source code comments.

------------------------------------------------------------------------

# JRE (Java Runtime Environment)

JRE provides an environment only to execute Java applications.

It consists of:

-   JVM
-   Base libraries
-   Integration libraries
-   User interface toolkits
-   Supporting runtime components

JRE does **not** contain development tools such as `javac`.

------------------------------------------------------------------------

# Relationship Between JDK, JRE, and JVM

``` text
JDK
 ├── JRE
 │     ├── JVM
 │     ├── Libraries
 │     └── Runtime Components
 │
 ├── javac
 ├── jar
 ├── javadoc
 └── Other Development Tools
```

------------------------------------------------------------------------

# Memory Areas Mentioned During JVM Execution

JVM uses different memory areas.

Two important ones are:

## Heap Memory

-   Stores objects and instance variables.
-   Shared among threads.
-   Garbage Collector primarily works here.

------------------------------------------------------------------------

## Stack Memory

-   Stores method calls.
-   Stores local variables.
-   Each thread gets its own stack.
